2: Multiple return paths

So far we've worked with functions that have one return statement and therefore one flow of execution. We've also learned how to express conditional logic using if statements. When using if statements within functions, you can create multiple return paths or multiple ways of returning a value and stopping the flow of execution at various points. Let's look at an example.

The flip_coin() function below expects a float input value and can either return "Heads" or "Tails" depending on the value that's passed in:

xxxxxxxxxx
 
def flip_coin(heads_probability):
    if heads_probability >= 0.5:
        return("Heads")
    else:
        return("Tails")

Multiple return paths allows us to make our functions more robust and more general since we can write code that captures a wider range of outcomes. If each function could only contain one return path, then we'd have to write 2 functions to replicate the above behavior:

xxxxxxxxxx
 
def is_heads(heads_probability):
    if heads_probability >= 0.5:
        return("Heads")
def is_tails(heads_probability):
    if heads_probability < 0.5:
        return("Tails")

In the next step, we'll learn how to combine multiple return paths with optional arguments to make the tokenize() function more flexible.

3: Optional arguments

Once we clean up the story text, the process of tokenizing both the story text and the vocabulary is actually the same (splitting on the whitespace character). To make our code more reusable, we can technically use the tokenize() function to tokenize both the story and the vocabulary. The vocabulary list doesn't need cleaning, however, so we need a way to specify when we want the input string to be cleaned and only run the cleaning logic in those cases.

We can accomplish this by modifying the tokenize() function in a few ways. First, we can wrap the code that cleans the input string in an if statement so that the code is only run if the Boolean value, named clean, passed into the if statement evaluates to True. Then, we can add an argument, named clean, to the function definition for the tokenize() function. This way, whenever we want the input string to be cleaned, we just set clean to the Boolean value True and when we don't, we set clean to the Boolean value False.

One downside to adding more arguments is we, or anyone else who wants to use the function we wrote, now has to spend more time deciding the values for each argument. Right now, all of the arguments are required arguments since we must pass in a value for each one (or an error is returned). We can instead specify a default value for an argument that we want the Python interpreter to set if we don't specify a value for that argument when calling the function. When we call that function, we won't be required to pass in that parameter and the Python interpreter will use the default value instead.

In the following code block, we modified the function definition for the tokenize() function by adding a new parameter, named clean, and by specifying a default value for it: 

xxxxxxxxxx
 
def tokenize(text_string, special_characters, clean=False):

We aren't required to pass in a 3rd parameter and Python will default to setting clean to False. We can then use the same tokenize() function for both the vocabulary and the story text:

xxxxxxxxxx
 
# We want the input string cleaned so we set the 3rd parameter to `True`.
tokenized_story = tokenize(story_string, clean_chars, True)
# Since we didn't pass in a value for the 3rd parameter, `clean` will be set to `False` by default.
tokenized_vocabulary = tokenize(vocabulary, clean_chars)

We're not done however! Even though we made the clean parameter optional, we need to still update the code in our function body to incorporate the clean parameter. By default, we don't want text_string to be cleaned. We therefore need to check if clean is True and only clean text_string if it is.

4: Named arguments

So far, we've been passing in arguments to functions we've written in the exact order we specified in the function definition. These are known as positional arguments since the Python interpreter uses the positions of the arguments to assign each value to the specific variable inside the function. As we increase the number of arguments a function can take, especially optional arguments, remembering the exact order of the arguments can become cumbersome. To help with this, the Python interpreter allows you to pass in named arguments in any order, which allows us to be more explicit in assigning values to the variables we want inside the function. When you call a function, you assign each value to the variable from the function definition.

Under the hood, Python uses a dictionary to map the argument names to the values we want when we call the function. Then, it uses that dictionary to make the values we passed in available within the function. Let's see what this looks like in code:

xxxxxxxxxx
 
# All 3 of these statements assign the same values to the function arguments.
tokenized_story = tokenize(clean=True, text_string = story_string, special_characters = clean_chars)
tokenized_story = tokenize(text_string = story_string, clean=True, special_characters = clean_chars)
tokenized_story = tokenize(special_characters = clean_chars, text_string = story_string, clean=True)

Since optional arguments have default arguments, this means we can leave out the clean parameter altogether. 

xxxxxxxxxx
 
tokenized_story = tokenize(text_string=story_string, special_characters=clean_chars)

In later courses, we'll be working with external libraries that have functions containing 10 or more optional arguments. The authors of these functions took advantage of named arguments and optional arguments to create functions that can flexibly work in many use cases.

The one caveat is that you can't have named parameters before positional arguments. When calling functions, you should specify the positional arguments first and then the named arguments.

6: Types of errors

As we start to take bigger advantage of functions to organize our code, our code can become more complex and we need to better understand the kinds of mistakes we can make when writing our code. We've briefly talked about errors in past missions, which are mistakes in our code that prevent our code from working as we expect.

The 2 main types of errors are:

  • syntax errors,
  • runtime errors.

Before code can be run, it must be parsed by the Python interpreter and organized into a data structure that represents the flow and complexity of the code you wrote. If any code is encountered that doesn't adhere to Python's language rules, the parsing halts and an error is returned. Rich coding environments like Atom and IPython Notebook help you prevent syntax errors, by performing syntax highlighting as you write code. Some examples of syntax errors include:

  • missing ending quotes for starting quotes,
  • using improper indentation,
  • using improper keywords.

Runtime errors occur only when the code is actually being run, which makes them harder to catch and prevent beforehand. Some examples of runtime errors include:

  • calling a function before it's defined,
  • calling a method or attribute that the object doesn't contain,
  • converting a value to a different type that can't be converted to that type.

In the rest of this mission, we'll explore some more specific examples of both of these errors.

8: Runtime errors

Runtime errors are the most common type of errors people make when writing code in any language. Runtime errors are more difficult to prevent than syntax errors since they occur when your code is running and can't be detected during parsing. The other challenge is that you often write code that works in predictable, normal situations but fails at handling a case you didn't account for. As you become more proficient in programming, you learn to identify potential runtime errors while you're writing the code itself and prevent them from happening altogether. Python, and most programming languages, contain a number of tools like error handling and automated tests that help you manage and reduce the runtime errors that occur. As the code in your projects become more complex, you'll learn how to incorporate your own errors into functions you write to gracefully fail and prevent certain negative behavior from occurring.

The documentation for Python 3.5 contains a full list of possible errors. If you glance at the hierarchy of errors here, the SyntaxError class is a small fraction of the entire tree of possibilities. The rest of the errors are runtime errors. Let's now dive into some examples of runtime errors!

9: TypeError and ValueError

In the following code, we try to concatenate a string to an integer. This returns a TypeError since the addition operator expects the values that are being added to be the same type.

xxxxxxxxxx
 
forty_two = 42
forty_two + "42"

This returns the following error:

You may have noticed that runtime errors look a little bit different. The error is displayed twice (TypeError is in the top left and bottom left corners). In addition, the text Traceback (most recent call last) is displayed on the top right corner. The traceback displays the code that was executed leading upto the most recent call that actually caused an error to be returned. In this case, our code was simple but we'll explore some errors where the traceback is more useful later in this mission.

Another common runtime error is the ValueError, which is returned when the type is correct but the value is still improper. A ValueError is returned when you try to convert a string representing a non-numeric value into a into a numeric type like a float. To cast, or convert, a value to a float, recall that you use the float() function:

xxxxxxxxxx
 
float("guardians")

Here's the error that's returned:

While trying to cast a string to a float isn't automatically an issue (which is why there was no TypeError), the specific value that we tried to cast as a float was problematic. The float() function didn't know how to cast the string "guardians" into a float and it returned an instance of ValueError instead.

10: IndexError and AttributeError

Another common runtime error we'll discuss is the IndexError, which is returned when trying to access a value that's not in the list's index. Trying to access the 5th element in a list containing only 2 elements would return this error. Here's an example in code:

xxxxxxxxxx
 
lives = [1,2,3]
lives[4]

Here's the error that's returned:

Since there's no value at index 4, an instance of IndexError is returned along with an arrow pointing to the problematic line of code. When working with lists of potentially variable length, use the len() function to look up the number of elements in the list. When attempting to access an element in a list, you can compare the index value with the number of elements in the list to ensure you're accessing that it's in that list's index.

The last runtime error we'll explore is the AttributeError. This occurs when trying to call a method or attribute on an object that doesn't contain it. In the following code, we try to call the split() method on the File handler instance instead of using the read() method to read it into a string first:

xxxxxxxxxx
 
f = open("story.txt")
f.split(" ")

Here's the error that's returned:

TextIOWrapper is a built-in Python object that represents the File handler and doesn't contain the split() method. Since the Python interpreter couldn't find the split() method within the TextIOWrapper class, an instance of AttributeError was returned.
